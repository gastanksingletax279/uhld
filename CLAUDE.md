# UHLD — Claude Code Instructions

## Project Summary

UHLD (Ultimate Homelab Dashboard) is a self-hosted, plugin-driven homelab management dashboard. FastAPI backend + React/TypeScript frontend, deployed as a single Docker container. Full architecture is in [ARCHITECTURE.md](ARCHITECTURE.md).

## Tech Stack

- **Backend:** Python 3.12, FastAPI, SQLAlchemy async + aiosqlite, APScheduler, passlib[bcrypt], PyJWT, Fernet encryption
- **Frontend:** React 18, TypeScript, Vite, Tailwind CSS, Zustand, Recharts, lucide-react
- **Deployment:** Multi-stage Docker (node:20-alpine → python:3.12-slim), Docker Compose, k8s manifests

## Key Patterns

### Adding a New Plugin

1. Create `backend/plugins/builtin/{plugin_id}/` package with:
   - `__init__.py` (empty)
   - `plugin.py` — class `FooPlugin(PluginBase)` with all required attributes and methods
   - `api.py` — FastAPI `APIRouter` (referenced from `get_router()`)
   - `models.py` — plugin-specific DB models if needed
   - `schema.py` — Pydantic schemas

2. Create `frontend/src/plugins/{plugin_id}/`:
   - `Widget.tsx` — compact dashboard card (receives `summary` prop)
   - `View.tsx` — full plugin page

3. The registry auto-discovers the plugin. No manual registration needed.

### Plugin Contract

```python
class FooPlugin(PluginBase):
    plugin_id = "foo"
    display_name = "Foo Service"
    description = "Short description"
    version = "1.0.0"
    icon = "server"           # lucide icon name
    category = "monitoring"   # see PluginBase for categories
    poll_interval = 60        # seconds; 0 = no polling
    config_schema = { ... }   # JSON Schema with optional "sensitive": true on fields

    async def health_check(self) -> dict: ...
    async def get_summary(self) -> dict: ...
    def get_router(self) -> APIRouter: ...
```

### Sensitive Config Fields

Fields with `"sensitive": true` in `config_schema` are:
- Encrypted before storing in `PluginConfig.config_json` (Fernet/`ENCRYPTION_KEY`)
- Masked (`"***"`) when returned from GET endpoints
- Never logged

### Error Handling

Plugin failures must never crash the main app:
- Wrap all external calls in `try/except`
- Return `{"status": "error", "message": "..."}` from `health_check()` and `get_summary()`
- Dashboard continues rendering other widgets

### Auth

- JWT in httpOnly cookie (`access_token`)
- Use `Depends(get_current_user)` for read routes
- Use `Depends(require_admin)` for write/mutating routes

## Development

```bash
# Backend (dev)
pip install -r requirements.txt
uvicorn backend.main:app --reload --host 0.0.0.0 --port 8000

# Frontend (dev)
cd frontend && npm install && npm run dev
# Proxies /api/* to localhost:8000

# Create first user

# Docker build
./build-run.sh
```

## Environment Variables

| Variable | Required | Description |
|---|---|---|
| `JWT_SECRET` | Yes | JWT signing secret |
| `ENCRYPTION_KEY` | Yes | Fernet key for plugin config encryption |
| `DATABASE_PATH` | No | SQLite path (default: `/data/uhld.db`) |
| `TZ` | No | Timezone (default: `America/Montreal`) |
| `LOG_LEVEL` | No | Python log level (default: `INFO`) |

Generate keys:
```bash
python -c "import secrets; print(secrets.token_hex(32))"          # JWT_SECRET
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"  # ENCRYPTION_KEY
```

## Plugin Config Management

The "Clear Settings" button (in the Configure modal, config mode only) calls `POST /api/plugins/{id}/clear`, which:
1. Disables the plugin (stops scheduling, unmounts nothing — routes stay mounted until restart)
2. Sets `config_json = None` in the DB

After clearing, the plugin appears as disabled with no stored credentials. The user must re-enable and re-enter config.

## Multi-Instance Plugin Support (Future Architecture)

Currently each plugin is a singleton — one instance per `plugin_id`. Future work to support multiple instances (e.g. two Proxmox clusters, three Docker hosts):

**Proposed approach:**
- Add an `instance_id` concept (slug, e.g. `proxmox-home`, `proxmox-work`)
- `PluginConfig` table gains an `instance_id` column (unique with `plugin_id`)
- Registry maps `(plugin_id, instance_id)` → instance
- Routers mounted at `/api/plugins/{plugin_id}/{instance_id}/...`
- Frontend: plugin views accept `instanceId` route param; switcher in the UI

**Design rules to follow when adding new plugins:**
- All state must live on the plugin instance (no module-level globals)
- Config is fully self-contained in the instance — no shared singleton state
- This ensures multi-instance will be a registry/routing change, not a plugin rewrite

## Code Style

- Python: Black formatting, `from __future__ import annotations`, typed everywhere, async throughout
- TypeScript: strict mode, no `any` in plugin code
- All plugin API calls wrapped in try/except with graceful error returns
